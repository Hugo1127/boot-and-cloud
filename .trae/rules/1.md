## 一、严格禁止项

### ❶ 框架 / 依赖禁用

- 禁止直接引入**Spring、Spring Boot、Spring Cloud** 任何官方 / 衍生依赖，所有核心能力手写实现；
- 禁止使用第三方 IOC/AOP 框架（如 Guice、Google Guice 等），自研 IOC 容器与 AOP 代理机制；
- 禁止集成成熟中间件 / 组件，服务注册发现、负载均衡、熔断降级等能力需完全手写实现，不得使用 Eureka、Nacos、Sentinel 等现有组件。

### ❷ 开发规范禁用

- 禁止代码结构混乱，所有模块需遵循统一编码规范，核心逻辑必须添加**详细注释**（注释覆盖率≥70%）；
- 禁止无单元测试，所有核心功能、工具类必须编写测试用例，单元测试覆盖率≥80%；
- 禁止忽略性能优化模块，JVM 调优、多线程 / 锁优化模块需完整实现，不得删减核心功能与测试用例。

## 二、强制必须项

### ❶ 开发实现要求

- 必须**从零手写实现**所有核心逻辑，可参考开源框架底层原理，但禁止直接复制、抄袭现有开源项目代码；
- 必须遵循**模块化、可插拔**设计，每个子模块独立解耦，具备单独编译、运行、测试能力，模块间通过标准化接口交互；
- 必须实现**功能完整性**，覆盖 IOC/AOP、Spring Boot 自动配置、微服务核心能力、JVM 调优、并发优化全核心模块，无功能缺失。

### ❷ 交付物要求

- 必须提供**完整可运行示例**，为每个核心模块编写单独示例，同时提供端到端的微服务完整示例（demo-app），演示所有功能的实际使用；
- 必须提供**可视化架构文档**，包含项目整体架构图、模块依赖图、核心功能调用流程图（如 Bean 生命周期、AOP 代理、HTTP 请求处理等），图表需清晰标注核心节点与交互逻辑；
- 必须提供**面试相关交付物**，整理全模块高频面试考点总结，包含考点原理、项目实现方式、核心代码位置；
- 必须提供**性能调优报告**，包含 JVM 调优、线程池 / 锁优化的测试数据、调优方案、性能对比结果与最佳实践。

### ❸ 过程管理要求

- 开发过程中若无法一次性完成所有模块，必须编写**标准化 todo.md**，清晰记录已完成工作、待完成模块、待开发功能、未解决问题，确保其他开发 / 智能体能无缝承接后续工作。

## 三、Todo.md 编写规范（强制遵循）

为保证工作衔接的连贯性，todo.md 需按**固定结构**编写，确保信息清晰、无歧义，其他大模型 / 开发可直接基于此继续开发，具体结构要求如下：

### 1. 已完成工作

- 按**模块维度**梳理，明确每个模块的完成状态（100% 完成 / 核心功能完成）；
- 标注已完成功能的**可运行性**（是否可单独测试 / 运行、是否有示例 / 测试用例）；
- 记录已产出的交付物（如架构图、面试考点、调优数据等）。

### 2. 待完成模块

- 按**开发优先级**排序（高 / 中 / 低），优先完成核心模块（mini-spring-core → mini-spring-boot → 微服务核心 → 性能优化）；
- 每个模块标注**待开发功能点**，明确具体开发内容（如 “mini-spring-cloud-registry：实现服务注册 / 发现、心跳检测、服务拉取”）；
- 标注每个模块的**交付物要求**（如是否需要单独示例、测试用例、面试考点梳理）。

### 3. 待解决问题

- 记录开发过程中遇到的**未解决技术问题**，标注问题所属模块、问题描述、已尝试的解决思路；
- 记录**待优化点**，如已实现功能的性能瓶颈、代码冗余点、测试用例缺失点。

### 4. 后续开发建议

- 给出待开发模块的**核心实现思路**（如参考的底层原理、关键技术点）；
- 明确**开发规范衔接点**，如编码规范、注释要求、测试用例编写标准；
- 标注**依赖关联**，如待开发模块依赖的已完成模块、接口调用方式。

## 四、补充实施要求

1. 所有模块的**pom.xml**需独立配置，确保单独构建无依赖问题，父工程仅做统一版本管理；
2. 核心工具类、通用接口需抽离至公共模块，避免代码冗余，提高复用性；
3. 性能优化模块需提供**可复现的测试用例**，支持直接运行获取调优前后的性能数据；
4. 面试考点总结需**绑定项目代码**，明确每个考点对应的核心代码文件、行号与实现逻辑；
5. 所有可视化图表（架构图、流程图等）需提供**源文件 + 导出版**（如 DrawIO 源文件 + PNG/JPG 导出版），方便后续修改。
